## 二、核心模块设计

### 2.1 知识解析层

**功能**：从ATT&CK框架提取完整的战术(Tactic)和技术(Technique)信息

**输入源**：
- MITRE ATT&CK 官方数据 (JSON/STIX格式)
- ATT&CK Enterprise Matrix
- ATT&CK Mobile Matrix (可选)

**输出**：结构化的知识点列表，包含：
- 战术ID和名称
- 技术ID、名称、描述
- 子技术ID、名称、描述
- 检测方法、缓解措施

### 2.2 题目规划层

**功能**：根据知识点规划题目生成任务

**规划策略**：
```
每个技术点 × 7种题型 = 基础题目数量
├── 单项选择题: 2-3道/技术
├── 多项选择题: 1-2道/技术
├── 判断题: 1-2道/技术
├── 排序题: 1道/技术 (适用场景)
├── 填空题: 1-2道/技术
├── 问答题: 1道/技术
└── 场景题: 1道/技术
```

### 2.3 多模型协作生成层

**协作模式**：

#### 模式A：并行生成 + 投票选择
```
题目需求 ──┬──▶ 模型A生成 ──┐
           ├──▶ 模型B生成 ──┼──▶ 聚合投票 ──▶ 最优答案
           └──▶ 模型C生成 ──┘
```

#### 模式B：角色分工
```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   出题者     │────▶│   审核者     │────▶│   优化者     │
│  (模型A)     │     │  (模型B)     │     │  (模型C)     │
└──────────────┘     └──────────────┘     └──────────────┘
```

#### 模式C：难度分层
```
基础题 ──▶ 模型A (擅长基础知识)
进阶题 ──▶ 模型B (擅长深度分析)
场景题 ──▶ 模型C (擅长综合应用)
```

### 2.4 质量保证层

**质量检查项**：
1. **格式一致性**：检查JSON Schema合规性
2. **内容准确性**：交叉验证答案正确性
3. **覆盖完整性**：确保所有技术点都有对应题目
4. **难度合理性**：评估题目难度分布
5. **去重检查**：避免相似题目重复

### 3.2 题型编码规范

| 题型 | 编码 | 说明 |
|------|------|------|
| 单项选择题 | SC (Single Choice) | 4个选项，1个正确答案 |
| 多项选择题 | MC (Multiple Choice) | 4-6个选项，2-4个正确答案 |
| 判断题 | TF (True/False) | 正确/错误判断 |
| 排序题 | SQ (Sequencing Question) | 按正确顺序排列 |
| 填空题 | FB (Fill in Blank) | 填写关键术语 |
| 问答题 | SA (Short Answer) | 简要回答问题 |
| 场景题 | SC (Scenario) | 基于场景的综合题 |

## 四、多模型协作流程

### 4.1 生成流程

```
Step 1: 任务分发
├── 根据技术点分配生成任务
├── 每个技术点分配给至少2个模型
└── 记录任务分配状态

Step 2: 并行生成
├── 各模型独立生成题目
├── 使用统一的Prompt模板
└── 输出标准化JSON格式

Step 3: 结果聚合
├── 收集所有模型输出
├── 按技术点和题型分组
└── 去重和相似度检测

Step 4: 质量评估
├── 答案一致性检查
├── 格式验证
└── 内容审核

Step 5: 冲突解决
├── 答案不一致时触发仲裁
├── 可选择人工审核或第三模型仲裁
└── 记录决策过程

Step 6: 最终输出
├── 生成标准化数据集
├── 输出统计报告
└── 存储到数据库
```

### 4.2 Prompt模板设计

#### 出题Prompt模板
```
你是一位网络安全渗透测试专家，熟悉MITRE ATT&CK框架。请根据以下要求生成一道评估题目：

【知识点信息】
- 战术: {tactic_name} ({tactic_id})
- 技术: {technique_name} ({technique_id})
- 子技术: {sub_technique_name} ({sub_technique_id})
- 技术描述: {description}

【题目要求】
- 题型: {question_type}
- 难度: {difficulty}
- 考察重点: {focus_area}

【输出格式】
请严格按照以下JSON格式输出：
{
  "question": "题目内容",
  "options": {"A": "选项A", "B": "选项B", ...},
  "correct_answer": "正确答案",
  "explanation": "详细解析"
}

请确保：
1. 题目准确反映技术特点
2. 答案唯一且明确
3. 解析详细且准确
4. 选项具有合理干扰性
```

#### 审核Prompt模板
```
你是一位网络安全专家，请审核以下题目的质量：

【题目内容】
{question_data}

【审核要点】
1. 技术准确性：答案是否正确？
2. 题目清晰度：表述是否明确无歧义？
3. 选项合理性：干扰项是否合理？
4. 解析完整性：解析是否充分？

【输出格式】
{
  "is_valid": true/false,
  "score": 0-100,
  "issues": ["问题1", "问题2"],
  "suggestions": ["建议1", "建议2"]
}
```

## 五、技术实现建议

### 5.1 推荐技术栈

```
├── 数据获取
│   ├── mitreattack-python: ATT&CK数据解析
│   └── requests: API调用
│
├── 多模型调用
│   ├── LangChain: 模型编排
│   ├── OpenAI API: GPT系列
│   ├── Anthropic API: Claude系列
│   └── Google AI API: Gemini系列
│
├── 数据处理
│   ├── pandas: 数据处理
│   ├── pydantic: 数据验证
│   └── jsonschema: Schema验证
│
├── 质量保证
│   ├── sentence-transformers: 相似度计算
│   └── 自定义规则引擎
│
└── 存储
    ├── SQLite/PostgreSQL: 结构化存储
    └── JSON/JSONL: 数据导出
```

## 六、质量指标

### 6.1 覆盖率指标
- 战术覆盖率: 100%
- 技术覆盖率: 100%
- 子技术覆盖率: ≥80%

### 6.2 质量指标
- 答案准确率: ≥95%
- 格式合规率: 100%
- 模型一致性: ≥85%
- 去重率: ≥90%

### 6.3 多样性指标
- 题型分布均衡度: 各题型占比差异<10%
- 难度分布: 简单30%、中等50%、困难20%
